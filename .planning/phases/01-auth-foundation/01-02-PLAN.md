<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Interceptor → HTTP request | Untrusted: interceptor reads token from signal (trusted source), attaches to all outgoing requests |
| 401 response → Error interceptor | Untrusted: 401 could be from expired token, revoked token, or backend error; refresh flow handles all cases |
| Auth guard → Router | Untrusted: client-side guard is UX only, backend enforces permissions on every API call |
| Permission service → API response | Untrusted: permission data from API must be validated before use in route guards |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | Spoofing | error.interceptor.ts refresh flow | mitigate | BehaviorSubject-based queue in AuthService prevents concurrent refresh calls that would invalidate rotated tokens (single-use backend rotation) |
| T-01-06 | Information Disclosure | auth.interceptor.ts token in headers | mitigate | Token sent via Authorization header (not URL params or body); recommend HTTPS in production to prevent MITM interception |
| T-01-07 | Tampering | auth.guard.ts client-side check | accept | Documented limitation: client-side guards are UX convenience; backend enforces permissions on every API call. No sensitive data exposed on unauthorized pages. |
| T-01-08 | Information Disclosure | permission.service.ts permission data | mitigate | Permissions loaded per-user from authenticated API endpoint; backend filters by user_id, preventing access to other users' permissions |
| T-01-SC | Tampering | npm package installs | mitigate | All packages verified in RESEARCH.md Package Legitimacy Audit (all Approved, official Angular ecosystem) |
</threat_model>
