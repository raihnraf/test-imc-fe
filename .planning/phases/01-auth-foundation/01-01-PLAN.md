<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Backend API | Untrusted credentials sent over HTTP; tokens returned and stored client-side |
| sessionStorage → JavaScript | Refresh token in sessionStorage accessible to any script on the page (XSS vector) |
| In-memory signals → Component templates | Signals exposed as readonly; components cannot mutate token state directly |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | auth.service.ts token storage | mitigate | Use in-memory signals for access_token (not localStorage); refresh_token in sessionStorage only (cleared on tab close) |
| T-01-02 | Information Disclosure | auth.interceptor.ts (future) | mitigate | Token attached via Authorization header (not cookie), preventing CSRF; Bearer scheme requires HTTPS in production |
| T-01-03 | Spoofing | auth.service.ts refresh flow | mitigate | Backend rotates refresh tokens (single-use); frontend sends refresh_token once per refresh cycle, preventing replay |
| T-01-04 | Tampering | auth.guard.ts (future) | accept | Client-side guards are UX convenience only; backend enforces permissions on every API call (documented in threat model) |
| T-01-SC | Tampering | npm package installs | mitigate | slopcheck + blocking human checkpoint for [ASSUMED]/[SUS] — all packages in RESEARCH.md are official Angular ecosystem (Approved) |
</threat_model>
